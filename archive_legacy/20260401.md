# 2026-04-01 工作报告（uav_delta_capture 报错排查与系统落地）

## 一、背景

在主系统 Ubuntu 20.04 + ROS1 Noetic 环境下，通过 Docker 运行 ROS2 Humble 容器，目标是构建并启动无人机协同 Delta 机械臂抓取系统。工程目录位于 [uav_delta_capture](uav_delta_capture)。

## 二、问题现象与根因

### 1. `ros2: command not found`
- 现象：容器内/主机环境中直接执行 `ros2` 失败。
- 根因：未先加载 ROS2 环境，或在主系统（非 ROS2）中执行 ROS2 命令。
- 结论：需在 ROS2 容器中执行，并先 `source /opt/ros/humble/setup.bash`。

### 2. `Package 'uav_bridge' not found`
- 现象：`ros2 launch uav_bridge ...` 仅搜索 `/opt/ros/humble`，看不到工作空间包。
- 根因：未正确叠加工作空间 `install/setup.bash`，以及早期包类型识别异常（被识别为 `ros.catkin`）。
- 结论：包类型统一为 ament 后，全量构建并 source 工作空间即可。

### 3. `ros.catkin` 识别异常
- 现象：`colcon list` 中部分包显示为 `ros.catkin`。
- 根因：相关 `package.xml` 缺少 `build_type` 导出。
- 结论：在 `package.xml` 中补齐：
  - `<export><build_type>ament_cmake</build_type></export>`

### 4. `uav_bridge` 链接/构建失败
- 现象：
  - TF 变换相关 undefined reference（`tf2::doTransform<PointStamped>`）
  - CMake 报 target 重复（`uav_bridge_node` 重复定义）
- 根因：
  - `tf2_geometry_msgs` 依赖与头文件可见性问题
  - `CMakeLists` 出现重复 target 定义
- 结论：清理并规范 [uav_bridge/CMakeLists.txt](uav_delta_capture/src/uav_bridge/CMakeLists.txt) 后恢复正常。

### 5. 运行路径混用
- 现象：`/home/bo/...` 与 `/home/ros2_ws/...` 混用导致环境判断混乱。
- 根因：主机与容器内挂载路径认知不一致。
- 结论：固定在容器内单一路径构建与运行。

## 三、采取的修复动作

1. 修正包元信息，统一 ament 构建体系：
   - [uav_bridge/package.xml](uav_delta_capture/src/uav_bridge/package.xml)
   - [delta_kinematics/package.xml](uav_delta_capture/src/delta_kinematics/package.xml)
   - [uav_delta_msgs/package.xml](uav_delta_capture/src/uav_delta_msgs/package.xml)

2. 修正桥接包构建配置：
   - [uav_bridge/CMakeLists.txt](uav_delta_capture/src/uav_bridge/CMakeLists.txt)
   - 去重 `add_executable(uav_bridge_node ...)`
   - 保留 `tf2`, `tf2_ros`, `tf2_geometry_msgs` 依赖

3. 确保接口与节点工程可安装可启动：
   - 消息/服务：
     - [GraspTarget.msg](uav_delta_capture/src/uav_delta_msgs/msg/GraspTarget.msg)
     - [SetArmStatus.srv](uav_delta_capture/src/uav_delta_msgs/srv/SetArmStatus.srv)
   - 运动学节点：
     - [delta_kinematics_node.cpp](uav_delta_capture/src/delta_kinematics/src/delta_kinematics_node.cpp)
   - 视觉节点：
     - [perception_node.py](uav_delta_capture/src/perception_logic/perception_logic/perception_node.py)
   - 桥接节点：
     - [uav_bridge_node.cpp](uav_delta_capture/src/uav_bridge/src/uav_bridge_node.cpp)
   - 统一启动：
     - [uav_delta_system.launch.py](uav_delta_capture/src/uav_bridge/launch/uav_delta_system.launch.py)

4. 规范容器内流程（最终有效）：
   - 进入容器
   - `source /opt/ros/humble/setup.bash`
   - `colcon build --symlink-install`
   - `source install/setup.bash`
   - `ros2 launch uav_bridge uav_delta_system.launch.py`

## 五、当前系统形态（2026-04-01）

已形成一个可运行的“无人机 + Delta 机械臂 + 视觉”ROS2 协同原型系统：

- 感知层：`perception_logic` 提供图像输入与目标偏移输出，预留 `infer_npu(image)` 接口，支持后续 NPU 推理集成。
- 变换桥接层：`uav_bridge` 订阅飞控位姿与视觉目标偏移，基于 TF2 转换到机械臂坐标系。
- 执行计算层：`delta_kinematics` 基于参数化几何尺寸（`L1,L2,R,r`）做逆解并发布关节角。
- 接口层：`uav_delta_msgs` 提供抓取目标消息与机械臂状态服务接口定义。

## 六、经验与后续建议

1. 严格区分“主机 ROS1 环境”与“容器 ROS2 环境”，避免交叉 source。
2. 每次构建后必须 `source install/setup.bash` 再运行。
3. 若仍有包可见性异常，优先检查：
   - `colcon list` 类型是否为 ament
   - `AMENT_PREFIX_PATH` 是否包含本工作空间 install 前缀
4. 建议下一步增加：
   - MAVROS 断连降级策略
   - 启动前自检脚本（环境、包索引、话题可用性）
   - 机械臂状态机服务端实现（对接 `SetArmStatus.srv`）
